# 更新手順

[English](../../en/03_運用/04_更新手順.md) | [日本語](./04_更新手順.md)


## 対象

- Atrakta CLI を更新する利用者向け
- latest-only 方針のため、旧挙動互換を前提にしない

## 1. 個人利用（ローカル中心）

1. 新しい `atrakta` バイナリに置換
   - ソース管理運用の場合は、利用対象プロジェクト配下の Atrakta ソースで再ビルド:
     - `cd <project>/atrakta`
     - `go build -o ~/.local/bin/atrakta ./cmd/atrakta`
     - `hash -r`
2. `atrakta doctor` を実行
3. `atrakta init --interfaces <主利用インターフェース>` を1回実行（対話/非対話を統一）

## 2. チーム利用（CIあり）

1. 検証用リポジトリで更新後の `go test ./...` を確認
2. CI の Atrakta バージョン固定値を先に更新
3. 対象リポジトリで `atrakta doctor` を実行して整合性を確認
4. 代表ワークフローで `start` / `resume` / `gc --scope tmp` を確認

## 2.1 メンテナ向け配布アーティファクト生成

1. 通常運用:
   - `main` push で `.github/workflows/release.yml` が自動実行される
2. 手動フォールバック:
   - `./scripts/build_release_artifacts.sh`
3. 生成物:
   - `.tmp/release/v<version>/packages`
   - `.tmp/release/v<version>/checksums.txt`
4. 利用者は Releases の事前ビルド済みバイナリを利用（Go不要）

## 3. 長期運用・重要リポジトリ

1. 更新前に `.atrakta/events.jsonl` を退避
2. `atrakta migrate check` を実行
3. `atrakta doctor` -> `atrakta start` の順で再同期
4. 必要に応じて `atrakta gc --scope events` で提案確認

## 4. 非対話環境（CIなど）

- `start` / `init` / `resume` の終了コードを明示ハンドリングする
  - `4`: `NEEDS_INPUT`
  - `5`: `NEEDS_APPROVAL`
  - `6`: `BLOCKED`
- `ATRAKTA_STATUS_JSON=1` を有効化して機械可読処理に統一する
