# 設計原則

[English](../../en/01_overview/02_design_principles.md) | [日本語](./02_設計原則.md)


## 統合原則

- Fast Path First, Strict Path On Demand
- 通常は最速経路で処理し、曖昧・異常・閾値超過時のみ厳密経路へ自動昇格する。

## 実装ルール

- 実行時は軽量:
  - runtime metrics は記録のみ（低オーバーヘッド）。
  - 通常イベントは group commit + phase-end flush。
- 厳密判定は検証時:
  - CI で p95/p99・トークン予算・fast-path命中率を fail-closed 判定。
  - 障害注入テスト（クラッシュ相当・lock競合・途中中断）を常時実施。
- 厳密化の自動昇格:
  - wrapper は軽量 stamp を優先し、疑義時のみ deep check を実行。
  - start は snapshot fast gate を優先し、10分ごと/設定差分/managed drift 時のみ strict path へ昇格。
  - events は `BLOCKED` / `FAIL` を urgent 扱いで即 sync。
- トークン効率:
  - conventions は `index -> 関連節 -> 必要時のみ全文` の段階読込を固定。
- ユーザー公開設定は最小:
  - 必須フラグのみ公開し、細かな最適化は `.atrakta/runtime/meta.v2.json` で内部管理。

## 客観的な完成条件

- 実行時の体感性能劣化がない。
- CI で性能/安全/整合の回帰が自動検知される。
- 24h/72hソークで連続運用の劣化が出ない。
- ユーザーの追加操作が増えない。
