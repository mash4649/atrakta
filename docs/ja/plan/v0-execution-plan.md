# v0 実行計画

## 目的

Atrakta 書き換えのための、最小だが実装可能な契約ベースラインを構築する。
ベースラインは、手動セットアップなしでブラウンフィールドと新規プロジェクトのオンボーディングを支えなければならない。
運用の入口は `atrakta run` とする。
本書き換えは `v0.14.1` のコマンド／データ互換を要求しない。

## 譲れない原則

1. Detect -> Plan -> Apply の規律
2. 管理対象のみ破壊的変更
3. 曖昧なときは prune しない
4. 追記のみのイベント可観測性
5. 決定的リプレイと明示承認

## デリバリの形

- 契約アーティファクトは第一級でバージョン管理される。
- リゾルバの入出力は明示的に検査可能。
- 変更は常に提案ゲート付き。
- オンボーディングは `detect -> propose -> accept`。
- 管理ランタイムの書き込みは `detect -> plan -> apply`。
- 正規への書き込み経路は投影から切り離す。

## バージョニング方針

- 書き換えラインは `1.0.0-alpha.2` から始まる。
- 破壊的な契約変更はアルファ中もメジャーを上げる。
- 互換を保つスキーマ追加はマイナーを上げる。
- ドキュメントのみの更新はパッチを上げる。

## イシュー順序

1. イシュー 1: レイヤ境界契約
2. イシュー 2: ガイダンスの強さと優先順位
3. イシュー 3: 投影モデルと一方向ルール
4. イシュー 4: 失敗ルーティングと厳格な状態機械
5. イシュー 5: 管理スコープと変更ポリシー
6. イシュー 6: レガシー統治と昇格ルール
7. イシュー 7: オペレーション能力モデル
8. イシュー 8: 拡張境界と評価順序
9. イシュー 9: 監査の整合性と保持ポリシー
10. イシュー 10: Inspect / Preview / Simulate インターフェース

## ゼロ設定オンボーディングフロー

### フェーズ A: Inspect

- プロジェクトルートとアセットサーフェスを検出する。
- ツール、ワークフロー、指示、リスク候補を列挙する。
- 書き込みは行わない。

### フェーズ B: Classify

固定順でリゾルバを実行する:

1. `classify_layer`
2. `resolve_guidance_precedence`
3. `check_projection_eligibility`
4. `resolve_failure_tier`
5. `check_mutation_scope`
6. `resolve_legacy_status`
7. `resolve_extension_order`
8. `resolve_audit_requirements`

### フェーズ C: Propose

- オンボーディング提案バンドルを組み立てる。
- すべての書き込み操作は提案アーティファクトとして保持する。

### フェーズ D: Accept

- 受諾した提案だけを正規ストアと generated ストアに永続化する。

## 初期出力契約

すべてのリゾルバは次を返す必要がある:

- `input`
- `decision`
- `reason`
- `evidence`
- `next_allowed_action`

## 初期バックログ

1. コアおよび正規エンティティの最小スキーマ
2. 決定的 IO 契約を持つリゾルバスタブ
3. 契約テストとフィクスチャ
4. オンボーディング提案バンドルスキーマ
5. 安全なデフォルトのポリシープロファイル

## v0 ベースラインの完了条件

- スキーマの所有境界が文書化されている。
- 必須リゾルバ API がすべて明示的な IO 契約を持つ。
- 変更のみの経路が利用可能である。
- ブラウンフィールド向けの安全なデフォルトが定義されている。
- 契約およびリゾルバ向けの決定的テストフィクスチャが実行できる。
- リフレッシュスコープの概念カバレッジが文書化され、run 中心の E2E で検証される。
