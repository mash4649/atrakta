### ハーネス学習の反映（Anthropic, 2026-03-24）

参照: [Anthropic, "Harness design for long-running application development"](https://www.anthropic.com/engineering/harness-design-long-running-apps)

- ハーネスは大きければよいわけではない。モデル世代ごとに再計測し、本当に効く要素だけを残す。
- `resume` はコンテキスト窓の再利用ではなく、spec・checkpoint・next action を渡す handoff 設計として扱う。
- 生成と評価は分離する。アプリ品質を扱うタスクでは、schema 検証だけでなく実ブラウザや実行環境での evaluator が必要。
- 主観的な品質も、受け入れ条件と scoring rubric に落として実行可能にする。
- planner / evaluator / reset を全タスクに固定で使わず、効果が出る条件でだけ有効化する。

## 4. 技術的課題・検討事項

### 4.1 アーキテクチャ判断が必要な項目

| 検討事項 | 選択肢 | 判断基準 | 推奨 |
|----------|--------|----------|------|
| 旧 events.jsonl の扱い | (A) 読み取り専用の履歴データ / (B) 必要時のみ取り込み | v0 の所有権を保ち、結合を最小化する | **(A)**: 旧 `/.atrakta/events.jsonl` は読み取り専用とし、変換層は置かない |
| state スキーマの版管理 | (A) 0.14.1 互換サブセット / (B) v0 独自＋バージョンフィールド | 既存ユーザーの移行パス | **(B)**: `schema_version` フィールドで版管理し、migrate check で変換 |
| インターフェース解決の拡張性 | (A) ハードコード / (B) プラグイン式 / (C) バインディング定義ファイル | 新 IDE 追加のしやすさ | **(C)**: 現行の `adapters/bindings/*/binding.json` を拡張 |
| プロジェクション書き出し戦略 | (A) 全上書き / (B) マージ / (C) proposal-only | ユーザーのカスタマイズ保護 | **(C) → (B)**: 初期は proposal-only、承認後マージ |
| 外部依存の方針 | (A) 標準ライブラリのみ維持 / (B) 厳選ライブラリ許可 | ビルド速度、バイナリサイズ、セキュリティ | **(A)** を基本とし、正当な理由がある場合のみ (B) |
| CI/CD プラットフォーム | (A) GitHub Actions のみ / (B) マルチ CI 対応 | 普及対象のユーザー層 | **(A)** を優先し、需要に応じて (B) |

### 4.2 技術的負債・リファクタリング候補

| 対象 | 課題 | 対応案 |
|------|------|--------|
| `cmd/atrakta/cmd_run.go`（約 1200 行） | セッションライフサイクルと apply オーケストレーションが集中 | start / run / resume の共通ライフサイクルを小さな単位へ分割 |
| リゾルバの共通出力構造 | `resolvers/common/output.go` に集約途中 | 全リゾルバで統一的な出力インターフェースを徹底 |
| フィクスチャ管理 | `fixtures/` 配下の手動管理 | テーブル駆動テスト + フィクスチャ自動生成の検討 |
| 用語の揺れ | `follow-up-vocabulary-alignment.md` で認識済み | Phase 5 の用語集整備で解消 |

### 4.3 スケーラビリティ考慮

| シナリオ | 現状の制約 | 必要な対応 |
|----------|------------|------------|
| モノレポ（数千ファイル） | detect の全ファイルスキャン | インクリメンタル検出、.gitignore 連携、キャッシュ |
| 大規模チーム（10+ 人） | 単一 canonical / state | ロック機構、マージ戦略、チーム別スコープ |
| CI/CD パイプライン | `go run` 前提 | プリビルドバイナリ、Docker イメージ、キャッシュ戦略 |
| 複数 AI ツール同時利用 | インターフェース解決が単一 | マルチインターフェース並行対応の検討 |

## 6. リスクと緩和策

| リスク | 影響度 | 発生可能性 | 緩和策 |
|--------|--------|------------|--------|
| ランタイム統合の複雑化による開発遅延 | 高 | 中 | Phase 1 を最小スコープに絞り、段階的に拡張 |
| AI ツール側の API 変更による互換性破壊 | 高 | 高 | バインディング定義の抽象化、アダプタ層の疎結合維持 |
| 概念の複雑さによるユーザー離脱 | 高 | 中 | Progressive disclosure の徹底、デフォルトの安全性 |
| 古いモデル前提にハーネスが過剰適合する | 高 | 中 | モデル世代ごとにプロファイル / アブレーション評価を行い、不要な手順を削除 |
| UX や統合品質に対する評価の見落とし | 高 | 中 | 受け入れ成果物、ブラウザベース evaluator、スコア閾値を追加 |
