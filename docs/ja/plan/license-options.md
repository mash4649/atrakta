# ライセンス候補（提案）

## 目的

Atrakta の OSS ライセンスを、トレードオフを理解したうえで選定する。
本ドキュメントは代表的なライセンスを比較し、**オーナー決定待ち**の推奨案を示す。

## 範囲と前提

- Atrakta は個人・組織（商用利用を含む）での利用を想定する。
- 再配布・コントリビューションのフローは分かりやすいことを重視する。
- 企業導入では特許の安心感が重視され得る。

前提が崩れる場合、本ドキュメントの推奨は無効として再評価する。

## 比較対象

### ざっくり比較

| ライセンス | 類型 | 特許許諾 | コピーレフト | 主な義務 |
|---|---|---:|---:|---|
| MIT | パーミッシブ | 明示なし | なし | 著作権表示とライセンス文面の保持 |
| Apache-2.0 | パーミッシブ | **明示あり**（特許報復条項あり） | なし | NOTICE（ある場合）・ヘッダ保持、変更の明示 |
| MPL-2.0 | 弱いコピーレフト | 明示あり | **ファイル単位** | 改変した MPL 対象ファイルは配布時も MPL を維持 |

### MIT

- **最適化しているもの**: とにかく簡単で採用されやすい、義務が最小。
- **トレードオフ**: 特許許諾が明示されない（企業によっては懸念になりうる）。
- **向いている場合**:
  - ライセンス条文を最短・最小にしたい。
  - 特許ポリシーは明示しない（または別枠で扱う）方針。

### Apache License 2.0

- **最適化しているもの**: パーミッシブでありつつ、特に特許面の法的明確性を高める。
- **主な特徴**:
  - コントリビュータから利用者への **明示的な特許ライセンス付与**。
  - 特許報復条項（攻撃的な特許主張を抑制しやすい）。
  - 通知・変更明示などの義務が整理されている。
- **向いている場合**:
  - 商用導入を含む広い採用を狙い、法務レビューの摩擦を減らしたい。
  - 外部コントリビューションを見込み、特許のデフォルト方針を定めたい。

### Mozilla Public License 2.0（MPL-2.0）

- **最適化しているもの**: 「改善は共有してほしい」を促しつつ、プロジェクト全体の強いコピーレフトは避ける。
- **コピーレフトの単位**:
  - MPL 対象ファイルを改変して配布する場合、改変後もそのファイルは MPL を維持する必要がある。
  - 新規ファイルは（MPL のルールに従いつつ）別条件にでき、GPL のように全体へ波及しない。
- **向いている場合**:
  - 商用利用は可能にしつつ、改変の還流（特にファイル単位）を促したい。
  - 一部の導入先でコンプライアンス負担が増える点を許容できる。

## 実務上の含意（こちら側で何が変わるか）

### コントリビューション（inbound）運用

- **MIT**: 最もシンプル。リポジトリ運用で「同条件で受け入れる」方針を明確にすれば足りることが多い。
- **Apache-2.0**: コントリビュータが多いプロジェクトで一般的。必要なら `NOTICE` と相性が良い。
- **MPL-2.0**: ファイル境界の運用設計が重要。コントリビュータがファイル単位コピーレフトを理解している必要がある。

### 特許と企業導入

- **MIT**: 明示的保証が少なく、法務から追加確認が来る可能性がある。
- **Apache-2.0**: 明示的特許許諾のため、企業の法務レビューが通りやすい傾向。
- **MPL-2.0**: 特許条項はあるが、コピーレフトが敬遠されるケースがある。

### 互換性とエコシステムの期待値

- いずれも OSI 承認で広く認知されている。
- **Apache-2.0**: 企業向けのインフラ／CLI ツールで特に一般的。
- **MIT**: ライブラリや小規模ツールで非常に一般的。
- **MPL-2.0**: 強いコピーレフトは避けたいが還流は促したい、という場合に採用されやすい。

## 推奨（提案 — オーナー決定が必要）

### 既定案: Apache-2.0

**理由**:

- CLI/ツールとしての広い採用（商用含む）に合わせやすい。
- **明示的な特許許諾**により、組織導入の摩擦を減らせる。
- コピーレフトなしで通知・帰属義務が整理されている。

### 現実的な代替案

- **MIT**: 特許の明確性よりも、最短・最小のシンプルさを最優先するなら有力。
- **MPL-2.0**: 改変の還流（ファイル単位）を促したいなら有力（ただし導入負担は増えやすい）。

## オーナー判断のチェックリスト

- 特許許諾をデフォルトで明示したいか？（Yes なら **Apache-2.0** or **MPL-2.0**）
- 改変への還流要件を入れたいか？（Yes なら **MPL-2.0**）
- コンプライアンス負担を最小にしたいか？（Yes なら **MIT**）

## 次のステップ（オーナー決定後）

- 選定したライセンスに合わせてトップレベルに `LICENSE` を追加する。
- Apache-2.0 の場合、必要なら `NOTICE` を追加（または不要であることを確認）する。
- README / docs に選定ライセンスを簡潔に明記する。
